# AI 助手模块开发方案

> 适用于智习灵枢项目中基于 `spring-ai-alibaba` 的 AI 助手模块建设，覆盖小程序用户使用、后台管理、审计治理、偏好记忆和业务闭环。

## 1. 文档目标

本文档用于指导智习灵枢 AI 助手模块的设计与开发，目标不是做一个单纯的聊天窗口，而是做一个可预约、可管理、可审计、可运营的业务闭环系统。

本轮方案聚焦以下目标：

- 小程序用户可通过自然语言完成座位预约、取消预约、查询预约、查看学习统计
- AI 助手可理解并沉淀用户座位偏好与使用习惯，用于后续推荐
- 后台可配置模型、智能体、工具权限、审计记录、风控规则和运营指标
- AI 只做智能编排，不绕过现有预约业务规则
- 现有预约服务继续作为唯一业务真相源，避免 AI 直接操作核心数据

本文档同时明确一个重要边界：

- 本期不是建设一个通用 AI 平台，也不是重做一个 Dify
- 本期要做的是“能真正提升预约效率和体验的业务型 AI 助手”
- 业务上要有意内聚到智习灵枢预约场景
- 技术上不能把预约细节直接写死到平台核心

## 2. 现状与关键约束

结合当前仓库结构，存在以下关键前提：

- 现有预约主服务位于 `lingshu-study-reservoir-java/lingshu-module-reservation`
- 管理后台位于 `lingshu-vue3`
- 小程序位于 `lingshu-study-reservoir-mini`
- 代码仓库中已有 `yudao-module-ai` 相关菜单、表结构和模块痕迹，但尚未真正接入预约业务闭环
- 根聚合工程当前仍以 `Java 8 + Spring Boot 2.7.x` 为主
- `spring-ai-alibaba` 推荐基于 `JDK 17` 使用

因此本方案的核心结论是：

- 不建议直接把 AI 编排能力硬塞入当前预约主服务
- 建议新增独立 AI 编排服务，使用 `JDK 17 + Spring Boot 3.x + spring-ai-alibaba`
- AI 编排服务通过受控业务接口调用现有预约服务，避免技术栈冲突和业务侵入

### 2.1 定位修正

经过方案讨论后，本模块的定位进一步收敛为：

- 产品定位：预约业务型 AI 助手，不做通用型 AI 平台
- 首期目标：提高用户从“表达需求”到“完成预约”的效率
- 架构定位：高内聚于当前预约业务，低耦合于当前预约实现细节

这里需要明确区分两件事：

- 业务内聚是正确的，因为当前真实价值场景就是预约
- 技术耦合是不应该的，因为后续仍需要保留扩展到其他预约型业务的可能

因此，`lingshu-ai-assistant-service` 更适合作为“AI 助手运行时”，由 `reservation` 作为首个领域接入，而不是首期就做成一个泛化平台。

## 3. 建设原则

### 3.1 业务原则

- AI 不是新业务源，预约规则仍由预约服务决定
- AI 不是自动下单器，关键操作必须经过显式确认
- AI 不是纯问答机器人，要对预约转化率负责
- 偏好不是纯自然语言摘要，要结构化存储并可解释

### 3.2 技术原则

- 大模型只负责理解、规划、生成回复，不直接改业务表
- 实时可预约资源只能通过业务接口获取，不通过 RAG 猜测
- 对话状态、偏好画像、工具调用、审计日志要分层存储
- 智能体能力按工具白名单控制，不允许开放式越权调用

### 3.3 产品原则

- 优先做“预约闭环”，而不是大而全的 AI 平台
- 优先把高频场景做顺，再逐步扩展学习报告、运营推荐等能力
- 小程序端采用“对话 + 推荐卡片 + 确认提交”的混合交互，降低误操作
- 首期产品价值只围绕“更快预约、更少点击、更少失败”展开

## 4. 整体架构

### 4.1 推荐架构

```text
小程序 / 后台
    |
    v
AI 助手网关层
    |
    v
AI 编排服务（JDK17 + Spring AI Alibaba）
    |            |               |
    |            |               +-- 偏好画像服务
    |            +-- 会话与审计服务
    |
    +-- 业务工具调用适配层
                 |
                 +-- 预约服务
                 +-- 用户服务
                 +-- 统计服务
                 +-- 通知服务
```

### 4.2 服务职责划分

#### AI 编排服务

- 接收小程序或后台发起的 AI 请求
- 管理 Prompt、模型选择、工具调用和结构化输出
- 管理会话上下文、偏好总结、推荐理由和响应降级
- 记录每轮会话的输入、输出、工具链路和模型消耗

#### 预约服务

- 保持预约、取消、签到、签退、学习统计等业务规则不变
- 向 AI 暴露受控工具接口
- 作为最终业务执行服务，负责并发校验和事务一致性

#### 后台管理端

- 管理模型配置、智能体配置、工具配置、提示词版本
- 查询用户会话、偏好画像、失败原因、风控事件
- 审计 AI 是否存在误导、越权、错误推荐和高成本问题

### 4.3 工程落位建议

结合当前项目现状，建议采用以下目录落位：

| 工程/目录 | 建议动作 | 说明 |
|------|------|------|
| `lingshu-study-reservoir-java/lingshu-module-reservation` | 继续保留 | 作为预约业务真相源，不直接引入 `spring-ai-alibaba` |
| `lingshu-vue3` | 新增 AI 助手后台页面 | 模型、智能体、会话、画像、风控、运营 |
| `lingshu-study-reservoir-mini` | 新增 AI 助手用户页面 | 对话、偏好、历史、推荐卡片 |
| `lingshu-ai-assistant-service` | 建议新增独立工程 | 使用 `JDK 17 + Spring Boot 3.x + spring-ai-alibaba` |

如果本期不准备新开独立仓库，也建议至少新建独立服务目录，不建议直接混入现有 `Java 8` 聚合工程。

### 4.4 核心与业务边界

为了避免服务未来完全绑死在当前项目，建议按如下边界拆分：

#### Core 层

Core 层只负责通用运行时能力，不直接理解预约业务字段：

- 会话管理
- Prompt 管理
- 模型路由
- 工具注册与调用
- 结构化响应协议
- 审计日志
- 风控与确认机制
- 用户记忆框架

Core 层不应直接出现以下业务字段：

- `seatId`
- `classroomId`
- `bookingId`
- `bookingDraft`
- `reservationRule`

#### Reservation Domain 层

预约领域层负责当前项目的业务能力接入：

- 查教学楼 / 楼层 / 教室 / 座位
- 生成预约候选
- 生成预约草稿
- 确认预约
- 取消预约
- 基于偏好排序候选

这样做的目标不是“现在就做超级平台”，而是：

- 产品上，先把预约助手做深
- 代码上，不把预约实体写死进核心运行时

### 4.5 自研边界

本服务不建议与 Dify 做正面能力对标，也不建议首期做成通用 AI 开发平台。

本服务首期自研边界应限制为：

- 交易型预约编排
- 预约偏好记忆
- 预约相关审计和风控
- 预约结果卡片协议

以下能力不作为首期重点：

- 通用知识库平台
- 多业务线共享插件市场
- 复杂多智能体协作
- 泛化运营 Copilot
- 面向全公司的一站式 AI 平台

## 5. 功能范围

### 5.1 小程序端功能

### 5.1.1 AI 助手入口

- 首页新增 AI 助手入口
- 预约首页新增“智能预约”入口
- 我的页面新增“我的偏好”入口

### 5.1.2 对话预约能力

支持以下高频意图：

- 我要预约座位
- 帮我找明天上午有电源的靠窗座位
- 我想预约常去的 F2 2 楼
- 今天晚上还有空位吗
- 帮我取消明天下午那条预约

### 5.1.3 推荐与确认

AI 返回结果不能只有文字，必须支持结构化卡片：

- 预约候选卡片
- 替代方案卡片
- 我的预约卡片
- 偏好解释卡片

预约卡片至少展示：

- 日期
- 时间段
- 教学楼 / 楼层 / 教室
- 座位号
- 座位标签，如靠窗、电源、安静区
- 推荐原因
- 是否命中历史偏好
- “确认预约”按钮

### 5.1.4 偏好管理

小程序端提供两类偏好能力：

- 显式偏好：用户主动设置
- 隐式偏好：系统根据行为推断

显式偏好建议包含：

- 偏好楼栋
- 偏好楼层
- 偏好时间段
- 偏好时长
- 偏好座位特征
- 是否允许自动根据历史习惯推荐
- 是否允许 AI 在候选结果中自动排序

### 5.1.5 失败兜底

当 AI 无法成功完成预约时，前端要支持以下降级：

- 显示标准筛选页并带入 AI 已解析出的条件
- 提供备选时间段
- 提供备选教室
- 提供人工快捷入口，如“去手动选座”

### 5.2 后台管理功能

### 5.2.1 模型管理

- 模型供应商配置
- 模型 Key 管理
- 模型启停
- 模型限流、超时、重试策略
- 不同场景的模型路由配置
- 模型成本统计

### 5.2.2 智能体管理

- 智能体名称、角色、人设、描述
- 系统 Prompt 版本管理
- Prompt 草稿与发布
- 工具白名单配置
- 结构化输出模板配置
- 是否必须确认后才能执行工具

### 5.2.3 会话管理

- 会话列表查询
- 单轮消息查看
- 工具调用链路回放
- 响应耗时、token 消耗查看
- 失败原因分类
- 敏感问题检索

### 5.2.4 用户偏好画像管理

- 查看用户显式偏好
- 查看系统推断偏好
- 查看偏好来源和置信度
- 手动纠偏
- 禁止 AI 使用某些偏好字段

### 5.2.5 审计与风控

- 模型输入输出审计
- 敏感词与越权拦截
- 提示词注入防护日志
- 异常高频请求监控
- 异常预约失败监控
- 用户投诉与会话关联查询

### 5.2.6 运营分析

- AI 会话数
- AI 预约发起数
- AI 预约成功率
- 推荐点击率
- 候选卡片确认率
- 常见失败原因 Top N
- 用户满意度反馈
- 单用户平均调用成本

### 5.3 首期范围收敛

为了避免把项目做成“看起来很通用、实际难落地”的平台，本期范围明确收敛如下：

#### 本期必须做

- 对话预约
- 候选推荐
- 偏好记忆
- 确认下单
- 取消预约
- 我的预约查询
- 失败兜底与替代方案
- 后台基础审计

#### 本期可预留但不强做

- 学习统计问答
- 排行榜解释
- 成就解锁说明
- 公告和规则知识库
- 管理后台运营 Copilot

#### 本期不做

- 面向多项目的通用 AI 平台
- 可视化工作流编排平台
- 多租户插件市场
- 对外开放的通用 AI SDK

## 6. 核心业务闭环

### 6.1 预约闭环

```text
用户提问
 -> 意图识别
 -> 条件补全 / 必要澄清
 -> 查询实时资源
 -> 结合偏好排序候选
 -> 返回候选卡片
 -> 用户确认
 -> 调用预约服务正式下单
 -> 返回预约结果与后续动作
 -> 记录偏好与行为信号
```

### 6.2 取消闭环

```text
用户发起取消
 -> 识别待取消预约
 -> 校验是否可取消
 -> 二次确认
 -> 调用取消接口
 -> 返回取消结果
 -> 记录取消行为进入习惯分析
```

### 6.3 偏好闭环

```text
用户设置偏好 / 实际预约行为 / 取消行为 / 查询行为
 -> 生成结构化偏好信号
 -> 更新用户偏好画像
 -> 后续推荐时参与排序
 -> 展示推荐原因
 -> 用户可手动纠偏
```

### 7.3 行为信号建议

行为信号不要直接写进偏好表，建议单独沉淀：

- 查询了哪些条件
- 最终选择了哪个候选
- 是否接受推荐第一项
- 是否频繁取消
- 是否常在某时段签到
- 是否存在固定学习周期

这样可以支持后续偏好重算、模型优化和运营分析。

## 8. 工具调用设计

### 8.1 工具设计原则

- 工具必须窄接口、强约束、可审计
- 工具入参和出参必须结构化
- 工具执行前后必须落日志
- 写操作工具默认要求确认态

### 8.2 MVP 工具清单

| 工具名 | 说明 | 是否写操作 |
|------|------|------|
| `getCurrentMemberContext` | 获取当前用户基础上下文 | 否 |
| `getMemberPreferenceProfile` | 获取用户偏好画像 | 否 |
| `getReservationRules` | 获取预约规则 | 否 |
| `searchBuildings` | 查询教学楼 | 否 |
| `searchFloors` | 查询楼层 | 否 |
| `searchClassrooms` | 查询教室 | 否 |
| `getClassroomOpenSlots` | 查询教室可预约时间段 | 否 |
| `searchAvailableSeats` | 查询可用座位 | 否 |
| `rankSeatCandidates` | 按规则与偏好排序候选 | 否 |
| `createBookingDraft` | 生成预约草稿，不落正式订单 | 否 |
| `confirmBooking` | 确认预约 | 是 |
| `listMyBookings` | 查询我的预约 | 否 |

## 9. 对话与状态机设计

### 9.1 会话状态

建议使用显式状态，而不是全靠模型自由发挥。

| 状态 | 说明 |
|------|------|
| `INIT` | 会话初始化 |
| `UNDERSTANDING` | 解析用户意图 |
| `NEED_CLARIFICATION` | 缺失必要条件，需要补充 |
| `SEARCHING` | 正在查询资源 |
| `RECOMMENDING` | 返回候选方案 |
| `WAIT_CONFIRM` | 等待用户确认 |
| `SUBMITTING` | 正在提交预约 |
| `COMPLETED` | 已完成 |
| `FAILED` | 失败 |

### 9.2 必问条件

AI 不应该无限追问，只在缺失关键条件时提问。预约最低必需条件：

- 日期
- 起止时间
- 至少一个资源范围条件，或允许 AI 自动推荐

如果用户没有给楼栋或教室，可以允许 AI 按偏好自动补全。

### 9.3 回复输出格式

建议统一为结构化响应：

```json
{
  "replyType": "booking_recommendation",
  "text": "我帮你找到 3 个比较合适的座位。",
  "cards": [],
  "actions": [],
  "conversationState": "WAIT_CONFIRM",
  "toolLogs": []
}
```

前端据此渲染聊天气泡、候选卡片和操作按钮，而不是自行解析自然语言。

## 10. 偏好与记忆设计

### 10.1 记忆分层

建议拆成三层：

### 短期会话记忆

- 本轮会话已经提到的日期、时间、楼栋、偏好条件
- 生命周期仅限当前会话

### 长期结构化偏好

- 用户显式设置的偏好
- 行为分析沉淀出的稳定偏好
- 参与候选排序和推荐解释

### 长期文本摘要

- 用于辅助自然语言表达和对话连续性
- 不作为最终业务判断依据

### 10.2 偏好更新机制

建议采用“事件驱动 + 定时归并”的方式：

- 用户每次预约、取消、确认推荐、切换候选、手动筛选，产生行为信号
- 行为信号先写入 `ai_assistant_member_signal`
- 由定时任务或事件消费者统一归并到 `ai_assistant_member_preference`

这样可以避免实时链路过重，也便于回滚和重算。

### 10.3 推荐排序策略

候选排序建议使用组合评分：

```text
综合得分 = 业务可用性分 + 显式偏好匹配分 + 隐式偏好匹配分 + 历史接受率分 - 风险扣分
```

说明：

- 业务可用性分：完全满足预约规则和空闲条件
- 显式偏好匹配分：用户明确设置过的偏好
- 隐式偏好匹配分：行为推断出的稳定习惯
- 历史接受率分：用户过去对相似候选的接受倾向
- 风险扣分：近期高取消率、高爽约率等

## 14. 安全、审计与治理

### 14.1 风险点

- 模型幻觉导致错误承诺
- 提示词注入诱导越权
- 直接暴露敏感预约数据
- 高频自动重试导致成本失控
- AI 误解时间表达造成错误预约

### 14.2 防护要求

- 写操作必须带确认态和幂等控制
- 工具层做参数校验，不信任模型生成参数
- 用户身份必须从登录态读取，不能信任模型上下文里的用户编号
- 所有时间参数统一转标准格式
- 敏感字段脱敏后再入审计日志
- 会话日志支持按保留周期清理

### 14.3 审计记录要求

每轮会话至少记录：

- 用户输入
- 系统 Prompt 版本
- 模型版本
- 结构化输出
- 工具调用参数
- 工具返回结果摘要
- 最终执行结果
- 风险拦截标记
- token 消耗与耗时

## 15. 推荐技术选型

### 15.1 推荐实现方案

### AI 编排服务

- JDK 17
- Spring Boot 3.x
- `spring-ai-alibaba`
- Redis 用于短期会话上下文与限流
- MySQL 用于会话、偏好、审计持久化

### 业务调用方式

优先级建议如下：

1. 优先调用现有预约服务开放的内部 HTTP API
2. 如果后续预约服务升级到 JDK 17，再考虑合并成统一工程

### 15.2 为什么不建议首期做重度 RAG

本项目首期核心任务是“实时预约闭环”，而不是知识问答平台。

因此：

- 实时教室、座位、时间段、预约状态必须走业务接口
- RAG 仅适合接公告、预约规则、签到说明、常见问题等知识型内容
- 不能把实时资源查找交给知识库检索

### 15.3 为什么不建议首期做通用平台

如果首期直接把 `lingshu-ai-assistant-service` 做成面向所有业务的通用 AI 平台，会有以下问题：

- 价值验证点不清晰，容易长期停留在底座建设
- 团队会把精力投入到平台抽象，而不是预约体验提升
- 为了“通用”而过度设计，反而延长首期交付时间
- 最后得到的很可能只是一个能力缩水版 Dify

因此更合理的策略是：

- 业务上，聚焦当前预约场景
- 架构上，保留核心运行时与预约领域的边界
- 路线中，先用一个强场景验证，再考虑是否扩展

## 19. 当前实现建议结论

基于现有项目现状，本轮落地建议如下：

- 技术架构上，采用独立 AI 编排服务，而不是直接入侵当前预约主服务
- 产品范围上，先做预约闭环，不先做泛问答平台，也不先做通用 AI 平台
- 数据策略上，偏好必须结构化存储，文本总结仅作辅助
- 安全策略上，所有写操作必须确认并全链路审计
- 实施顺序上，优先小程序对话预约、偏好画像、后台审计，再逐步扩展
- 服务定位上，允许业务高内聚于当前预约场景，但不允许技术核心直接写死预约实体和流程

这套方案可以在不推翻现有预约系统的前提下，把 AI 助手做成真正可上线、可管理、可持续优化的业务能力。
